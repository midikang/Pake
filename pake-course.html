What Pake Does

From "Type a Command" to "Double-Click an App"

Turn Any Website Into a Desktop App

Imagine you love using WeRead to read books, but it only exists as a website. You have to open your browser, find the tab, and it gets lost among dozens of other tabs. You wish it was just... an app. Like Spotify or VS Code — something you can click to open, that lives in its own window.

That's exactly what Pake does.

Type one command — pake https://weread.qq.com — and out comes a standalone desktop app. Double-click it, and WeRead opens like any other application on your computer. No browser tabs. No bookmarks. Just... an app.

💡
Why Does This Matter?

As someone building with AI code tools, knowing how "website-to-app packaging" works helps you make better decisions: Should I build a web app or a desktop app? Can I wrap my project in a native shell? What are the tradeoffs of Electron vs. Tauri?

Let's Trace What Happens When You Run the Command

You open your terminal and type: 

$ pake https://github.com --name GitHub

Under the hood, here's the journey that command takes:

1
The CLI Parses Your Command

Pake reads "GitHub" as the app name and "https://github.com" as the URL to package

2
It Fetches an Icon

Pake tries to download GitHub's logo from multiple icon services — if one fails, it tries the next

3
It Generates Configuration Files

Settings like window size (1200x780), app name, and icon are merged into JSON configs

4
It Builds a Rust Application

The Rust compiler turns source code into a native app with a built-in web browser

5
It Packages Everything

On macOS, creates a DMG installer. On Windows, an MSI installer

And that's it. Five minutes later (or 30 seconds on subsequent builds), you have GitHub.dmg in your current folder. Open it, and GitHub loads in its own window — no browser chrome, no tabs, just the website acting like a native app.

The Secret: It's Not Really Building a Browser

Here's the clever part: Pake doesn't bundle an entire web browser like Electron does. Instead, it uses your computer's existing browser engine:

🍎
macOS

Uses WebKit — the same engine Safari uses, already built into macOS

🪟
Windows

Uses WebView2 — Microsoft's Chromium-based webview, included in Windows 10/11

🐧
Linux

Uses WebKitGTK — WebKit for Linux, installable via package managers

💡
Key Insight: Leverage What's Already There

This pattern — using the platform's existing components instead of bundling your own — is everywhere in software. It's why Pake apps are ~20x smaller than Electron apps. When building with AI, always ask: "What can I use that's already on the user's system?" Not every problem needs a bundled solution.

What Actually Gets Built? Let's Look at the Config

When Pake builds your app, it creates a configuration file that tells the app what to do. Here's what that config looks like for our GitHub app:

CODE 
{
  "windows": [{
    "url": "https://github.com",
    "width": 1200,
    "height": 780,
    "hide_title_bar": false
  }]
}
PLAIN ENGLISH

Start of the configuration...

This app will have windows (could be multiple)...

Load GitHub's website when the app opens...

The window should be 1200 pixels wide...

...and 780 pixels tall (laptop-friendly size)...

Keep the title bar visible (show app name and close button)...

End of window config...

End of file.

That's it. When the app launches, the Rust code reads this file and says: "Okay, open a 1200x780 window and load https://github.com in the built-in browser." Everything else — scrolling, clicking links, logging in — works exactly like the website because it IS the website, just displayed in a dedicated window.

The Cast of Characters

Who does what when Pake builds your app

Four Players Work Together to Build Your App

Think of Pake like a film production. You (the user) give the script — "I want a GitHub app." Then four departments collaborate to make it happen:

📦

CLI (Director)

Takes your command, coordinates everyone else, makes final decisions about what gets built.

⚙️

Config System (Screenwriter)

Turns your options (name, size, icon) into detailed instructions the app can read.

🦀

Rust Core (Actor)

The actual app that runs on your computer — opens windows, loads websites, handles shortcuts.

🎨

Icon Pipeline (Set Designer)

Downloads and converts icons so your app looks professional on every platform.

Each one has a specific job. The CLI is written in TypeScript and runs on Node.js. The Rust Core is written in Rust and compiles to native code. Let's see how they communicate.

Each component has a clear responsibility. The CLI never touches the browser engine — that's the Rust Core's job. The Rust Core never downloads icons — that's the Icon Pipeline's domain. This separation makes the codebase easier to maintain and debug.

💡
Universal Principle: Separation of Concerns

This pattern — splitting responsibilities into focused roles — is one of the most important ideas in software engineering. Engineers call it separation of concerns. When you're steering AI to build something complex, tell it: "Create separate modules for X, Y, and Z — don't mix their logic." You'll get cleaner, more debuggable code.

What the CLI Actually Does: The Code

Let's peek at the TypeScript code that runs when you type pake [url]. This is the orchestrator — the piece that kicks off the whole process:

CODE 
program.action(async (url, options) => {
  const appOptions = await handleInputOptions(options, url);
  const builder = BuilderProvider.create(appOptions);
  await builder.prepare();
  await builder.build(url);
});
PLAIN ENGLISH

When the user runs the pake command...

First, clean up and validate their options (name, size, etc.)...

Figure out which platform we're on (Mac/Windows/Linux) and create the right builder...

Install any missing dependencies (Rust, npm packages)...

Actually build the app...

Done!

Notice how compact this is. The CLI doesn't know how to build for macOS vs. Windows — it just says "create a builder" and the BuilderProvider picks the right one. That's another pattern worth knowing: the factory pattern.

Where to Find Each Component in the Codebase

If you clone the Pake repository, here's where each character lives:

📦
/bin/ — CLI (TypeScript)

Files like cli.ts, builders/MacBuilder.ts, options/icon.ts

⚙️
/src-tauri/*.json — Config Templates

pake.json, tauri.conf.json — merged with your options at build time

🦀
/src-tauri/src/ — Rust Core

lib.rs (setup), app/window.rs (WebView), app/invoke.rs (commands)

🎨
/bin/options/icon.ts — Icon Pipeline

580 lines of icon downloading, converting, and platform-specific formatting

Now when you want to change something — like "I want to add support for a new icon service" — you know exactly where to look: /bin/options/icon.ts. Or "I want to add a new keyboard shortcut" → /src-tauri/src/inject/event.js.
